# 05 — Валидаторы и Quality Gates

## 1. Главная задача

AI умеет красиво писать, но красиво ≠ полезно.
Поэтому каждый результат проходит через Generator → Validator → Quality Gate → Client.

## 2. Уровни валидации

### Level 1 — Schema Validation

- Проверка JSON
- Проверка типов данных
- Проверка обязательных полей
- Проверка структуры

### Level 2 — Business Validation

- Проверка маркетинговой логики
- Связность выводов
- Непротиворечивость

### Level 3 — Content Validation

- Качество контента
- Отсутствие воды
- Конкретность

### Level 4 — Execution Validation

- Выполнимость задачи
- Наличие времени выполнения
- Понятность инструкций

### Level 5 — Client Validation

- Поймёт ли клиент
- Не требует ли задание маркетолога
- 15-летний подросток должен понять

## 3. Stop Words Engine

Файл: `shared/constants/stop_words.json`

### Категория 1 — Маркетинговая вода

Запрещено: `уникальный`, `инновационный`, `революционный`, `синергия`, `экосистема`, `максимальная эффективность`, `лидер рынка`, `экспертный подход`, `комплексное решение`

Причина: не несут пользы.

### Категория 2 — Пустые советы

Запрещено: `развивать соцсети`, `повышать узнаваемость`, `работать над брендом`, `увеличивать вовлечённость`, `улучшать маркетинг`

Причина: нет действий.

### Категория 3 — Заглушки

Запрещено: `нет информации`, `не найдено`, `данные отсутствуют`, `требуется проверка`, `ручная проверка`

Причина: клиенту бесполезно.

### Категория 4 — Галлюцинационные формулировки

Запрещено: `гарантированно`, `100%`, `точно получите`, `обязательно вырастет`, `увеличится в X раз`

Причина: юридические риски.

## 4. Anti-Hallucination Engine

### Главное правило

Нельзя выдумывать факты.

### Если данных недостаточно

Разрешено:

```json
{ "confidence": "low", "assumption": "", "verification_method": "" }
```

### Запрещено утверждать без источника

- Рыночные цифры
- Проценты
- Конверсии
- Объём рынка
- Позиции конкурентов

### Fact Registry

Хранит подтверждённые факты:

```json
{ "fact": "У компании 7 залов", "source": "brief", "confidence": "high" }
```

Модель имеет право ссылаться только на факты с confidence: `high` / `medium`.

## 5. Repetition Validator

Проверяет: темы, CTA, офферы, Reels, Hooks, Stories, объявления.

### Правило

Повторяемость: ≤ 20%.

Если больше → FAILED.

### Content Archetypes (минимум 10)

`tour`, `before_after`, `checklist`, `case`, `faq`, `objection`, `mistake`, `behind_the_scenes`, `comparison`, `review`

Правило: один archetype нельзя использовать подряд.

## 6. Hook Validator

Запрещено (больше 2 раз за проект):

- «5 советов...»
- «3 причины...»
- «Как выбрать...»
- «Что нужно знать...»

## 7. CTA Validator

Запрещено повторять на протяжении 30 дней:

- «Напишите в директ»
- «Напишите в личку»
- «Оставьте заявку»

Минимум 10 CTA-механик: Слово-ключ, Чек-лист, Калькулятор, Бронь, Подбор, Консультация, Гайд, Подборка, Разбор, Тест.

## 8. Pain → Offer Validator

Проверка связи: Боль → Оффер.

Если оффер не решает боль → FAILED.

Пример:

- Боль: «Боюсь переплатить» + Оффер: «Скидка 10%» → PASS
- Боль: «Боюсь переплатить» + Оффер: «Подписывайтесь на Telegram» → FAIL

## 9. Content Logic Validator

Проверка цепочки: Avatar → Pain → Content → Offer → CTA.

Если цепочка разорвана → FAILED.

## 10. KPI Validator

Запрещено: `много`, `мало`, `хорошо`, `плохо`, `высокий CTR`.

Разрешено: `CTR > 2%`, `10 заявок`, `5 продаж`.

Все KPI должны быть числовыми.

## 11. Advertising Validator

Запрещено: бюджет 5000 ₽+ с первого дня.

Разрешено: 500 ₽, 1000 ₽, тест.

Проверка: Тест → Анализ → Масштабирование.

## 12. Sales Validator

Запрещено: давление, манипуляции, ложный дефицит, обман, гарантии.

Каждый сценарий обязан иметь цель и следующий шаг.

## 13. Execution Validator

Проверка: можно ли выполнить задачу?

Плохо: «Улучшить позиционирование» → FAIL.
Хорошо: «Откройте сайт. Измените заголовок на ...» → PASS.

## 14. Time Validator

Каждая задача обязана иметь время выполнения.

Пример: 10 минут, 20 минут, 45 минут.

Если нет → FAILED.

## 15. Client Simplicity Validator

Проверка: поймёт ли владелец бизнеса?

Если задача требует маркетолога, таргетолога, аналитика, копирайтера → FAIL.

Задача должна быть понятна 15-летнему подростку.

## 16. Dashboard Validator

Проверка HTML:

- Есть вкладка «Сегодня»
- Есть «Контент»
- Есть «Продажи»
- Есть «Реклама»
- Есть «Метрики»

Если нет → FAILED.

## 17. Dead Content Validator

Проверка: не является ли контент бесполезным.

Плохо: «Сегодня расскажем о нашей компании» → FAIL.
Хорошо: «3 ошибки при выборе фотостудии» → PASS.

## 18. Mission Validator

Каждая миссия должна вести к: Подписчику, Лиду, Продаже, Повтору, Рекомендации.

Если миссия не ведёт к бизнес-цели → FAILED.

## 19. Export Validator

Перед созданием HTML. Проверить:

- Все блоки PASSED
- Нет пустых данных
- Нет стоп-слов
- Нет заглушек
- Нет FAIL

Только тогда: EXPORT ALLOWED.

## 20. Final Quality Score

```json
{ "quality_score": 0 }
```

Формула: Schema + Business + Content + Execution + Client.

Шкала:

- 95–100: Production Ready
- 85–94: Good
- 70–84: Needs Review
- <70: Failed

## 21. Cross-Block Validators

### Validator 1: Avatar → Pain

Каждая боль принадлежит аватару. Иначе → FAILED.

### Validator 2: Pain → Offer

Каждая боль имеет оффер. Иначе → FAILED.

### Validator 3: Offer → CTA

Каждый оффер ведёт к действию. Иначе → FAILED.

### Validator 4: Content → Funnel

Каждый контент ведёт дальше по воронке. Контент ради контента → FAILED.

### Validator 5: Reels → Product

Каждый Reels связан с продуктом. Иначе → FAILED.

### Validator 6: Ads → Audience

Каждая реклама имеет конкретный сегмент. Иначе → FAILED.

```
├── sales_validator.py         # Скрипты продаж
├── execution_validator.py     # Выполнимость задач
├── client_simplicity.py       # Понятность клиенту
└── dashboard_validator.py     # HTML Dashboard
```
